🎯 標準化プラットフォームの概要

核心メッセージ

既存基幹システム（WMS/TMS）は変更不要。
外側に標準化レイヤーを追加して荷主連携を統一します。

このプラットフォームの位置づけ

  • 本システムは既存のWMS・TMS・基幹システムを置き換えるものではありません
  • 既存システムの外側に配置し、荷主連携・イベント追跡・障害検知・再処理を標準化する「連携アダプタ層」です
  • 各荷主の既存連携方式（API / EDI / ファイル）をそのまま受け入れ、内部で統一的に処理します

現状の課題 (AS-IS)

各荷主が異なる連携方式を使用しており、個別対応が必要です。障害発生時の可視性がなく、対応が遅延します。

荷主 AS-IS 連携方式 AS-IS の課題 TO-BE（標準化後）
東京物流 (SHP001) API直接連携（REST） APIバージョン変更時に個別対応が必要。エラー発生時の追跡困難。 → 統一APIゲートウェイ経由
大阪運輸 (SHP002) JX手順EDI EDI手順書の維持・管理コストが高い。障害検知が遅延。 → 統一EDIアダプタ経由
名古屋流通 (SHP003) FTPファイル転送 ファイル形式の不統一。転送失敗の検知が困難。 → 統一ファイルアダプタ経由

AS-IS: 連携方式だけでなく、確認・監視・再処理も荷主ごとに個別化

東京物流（API連携）

  • 認証設定: 個別
  • 疎通確認: 個別手順
  • 監視: 個別アラート
  • 再処理: APIログ確認後に手動対応
  • 仕様: 個別API仕様書

大阪運輸（EDI連携）

  • 未着確認: 運用担当が手動確認
  • 再送依頼: メール/電話
  • 取込確認: バッチログ確認
  • 再処理: 手順書ベース
  • 仕様: EDIファイルレイアウト資料

名古屋流通（ファイル連携）

  • 受信方法: SFTP/共有フォルダ
  • 文字コード/項目差異: 個別調整
  • 監視: ファイル未着監視が個別
  • 再投入: 手作業
  • 仕様: Excel/メールで管理

標準化プラットフォーム (TO-BE)

提供価値

  • 対応チャネル: API連携 / EDI連携 / ファイル連携（3チャネル対応済み）
  • 統一ゲートウェイ: 全荷主が統一されたゲートウェイを通じて接続
  • 集中追跡: 連携イベントの集中追跡・監視（連携追跡番号で全履歴を把握）
  • 障害検知・再試行: 障害発生時の自動検知と再試行機能
  • オンボーディング高速化: 新規荷主のオンボーディング時間を大幅短縮
  • 既存システム保護: WMS/TMSは一切変更不要
  • 拡張可能性: 今後、MQ連携（Kafka/RabbitMQ）やWebhook通知にも拡張可能

⚠️ 連携エラー再現

未登録の荷主コードや不正な出荷指示を送信し、連携エラー発生時にどの荷主・拠点・出荷指示が影響を受けているかを確認するデモです。

  • 登録されていない荷主コード（例: UNKNOWN_SHP）で出荷指示を送信すると、連携エラーが発生し、影響を受ける荷主・拠点・出荷指示が即座に表示されます
  • 未登録の拠点コードで出荷指示を送信した場合も、連携エラーとして検知され、どの拠点が見つからなかったかが表示されます
  • 連携エラーが発生したイベントの「再試行」ボタンで、エラー → 受付済 への状態回復を確認できます
